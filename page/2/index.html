【阅读笔记】rocketmq 命名服务 namesrv （三）

发表于 2019-05-04 | 更新于 2019-06-15

基本功能

  1. 生产者、消费者从 namesrv 中获取可用的 broker 地址（查询功能）

  2. broker 定时向 naemsrv 发送心跳信息，维护可用 broker 地址（更新信息）

  3. 保存 key-value 配置信息

多个 namesrv 实例组成集群，但相互独立，没有信息交换

命令参数

usage: mqnamesrv [-c ] [-h] [-n ] [-p]

-c,–configFile Name server config properties file
设置配置文件位置

-h,–help Print help
打印帮助

-n,–namesrvAddr Name server address list, eg: 192.168.0.1:9876;192.168.0.2:9876
设置服务器 IP 和端口

-p,–printConfigItem Print all config item
仅打印配置信息

功能实现

namesrv 的入口类是 org.apache.rocketmq.namesrv.NamesrvStartup， 主要功能由 NamesrvController 类组合起来，配置由 NamesrvConfig 和 NettyServerConfig 两个类保存

启动

  1. 创建一个 NamesrvController 对象

    创建 NamesrvConfig 和 NettyServerConfig 两个配置对象，如果指定了配置文件，就用配置文件中读取属性

    配置日志配置文件为 ${ROCKETMQ_HOME}/conf/logback_namesrv.xml

  2. 调用 controller 的 initialize 方法

  3. 注册 ShutdownHook，这个 shutdown hook 调用 controller 的 shutdown 方法

  4. 调用 controller 的 start 方法启动 namesrv

启动后打印 “The Name Server boot success” 后退出主线程

命令行参数的定义和解析由 srvutil 包的 ServerUtil 类实现， ServerUtil 是对 commons-cli 的一层封装

定时任务

所在的类 执行器 频率 作用
NamesrvController scheduledExecutorService 10s routeInfoManager.scanNotActiveBroker()
10 min 执行 kvConfigManager.printAllPeriodically()，在日志文件打印 KVConfigManager 的配置内容

保存 key-value

key-value 配置由 KVConfigManager 类管理

KVConfigManager 的成员有
ReadWriteLock lock：一个读写锁，保护 configTable 成员

HashMap<String, HashMap<String, String>> configTable：一个二维的散列表，第一个 key 是 Namespace，第二个 key 是 Key，之后是 value

成员方法有
load：从文件加载 configTable，文件路径由 NamesrvConfig 的 kvConfigPath 配置，默认是 ${user.home}/namesrv/kvConfig.json

persist：将 configTable 保存到文件，json 格式

putKVConfig：增加一个 key-value 配置

deleteKVConfig：删除一个 key-value 配置

getKVListByNamespace：获取配置

getKVConfig：获取配置

printAllPeriodically：周期性地打印配置的内容

保存 broker 信息

broker 信息由 RouteInfoManager 类管理

RouteInfoManager 的成员有

ReadWriteLock lock：一个读写锁，保护所有数据成员

HashMap<String, List> topicQueueTable：保存 topic 信息，key 是 topic 名，QueueData 包含了队列信息。

HashMap<String, BrokerData> brokerAddrTable：保存 broker 信息，key 是 broker name，BrokerData 包含了 broker 信息

HashMap<String, Set> clusterAddrTable: 保存 broker 集群信息，key 是集群名， set 保存了 broker namesrv

HashMap<String, BrokerLiveInfo> brokerLiveTable：保存 broker 的存活信息，key 是 broker 的地址

HashMap<String, List> filterServerTable：保存 filter server，key 是 broker 的地址，list 里是 filter server

功能实现

namesrv 主要接收请求 ，不会发起请求，请求由 DefaultRequestProcessor 类处理

命令 作用
PUT_KV_CONFIG 调用 KVConfigManager 成员方法增加 key-value 配置
GET_KV_CONFIG 获取配置
GET_KVLIST_BY_NAMESPACE 获取配置
DELETE_KV_CONFIG 删除配置
UPDATE_NAMESRV_CONFIG 更新 namesrv 的配置
GET_NAMESRV_CONFIG 获取 namesrv 的配置
QUERY_DATA_VERSION
REGISTER_BROKER broker 上线时，以及定期调用，用于注册 broker 信息
UNREGISTER_BROKER broker 正常下线时调用，更新 broker 信息
GET_BROKER_CLUSTER_INFO
WIPE_WRITE_PERM_OF_BROKER
GET_ALL_TOPIC_LIST_FROM_NAMESERVER
DELETE_TOPIC_IN_NAMESRV
GET_TOPICS_BY_CLUSTER 获取一个集群的所有 topic
GET_SYSTEM_TOPIC_LIST_FROM_NS
GET_UNIT_TOPIC_LIST
GET_HAS_UNIT_SUB_TOPIC_LIST
GET_HAS_UNIT_SUB_UNUNIT_TOPIC_LIST
GET_ROUTEINTO_BY_TOPIC 根据 topic 获取 broker 地址

register broker

broker 发送 REGISTER_BROKER 命令时，会带来brokerName、brokerAddr、clusterName、haServerAddr、brokerId 以及 topic 信息。这些信息会保存到 RouteInfoManager 中

  1. 上锁

  2. 修改 clusterAddrTable，需要 clusterName 和 brokerName 两个要素

  3. 修改 brokerAddrTable，由于 BrokerData 里保存了 brokerId，当发生主从切换时，要先删除原 slave 的 brokerData，再插入 新 master 的 BrokerData

  4. 如果 broker 是 master 并且带来了 topic 的信息， 则更新 topicQueueTable

  5. 更新 brokerLiveTable 的数据

  6. 更新 filterServerList

  7. 解锁

返回的结果里会包含这组 broker 的 master 地址以及 master 的 HA 地址

get route info by topic

从 topicQueueTable 中获取 QueueData 列表，然后重组成 TopicRouteData 数据

Broker 存活检查

存活检查有两种方法

  1. NamesrvController 有个线程定期（10秒）调用 routeInfoManager.scanNotActiveBroker()，检查brokerLiveTable 中的 lastUpdateTimestamp。
    如果 lastUpdateTimestamp + BROKER_CHANNEL_EXPIRED_TIME （2分钟）< 当前时间，则认为这个 broker 已经不再存活，并将之从 brokerLiveTable 删除。而 broker 周期性发送 REGISTER_BROKER 命令到 namesrv，namesrv 这时会更新 lastUpdateTimestamp

  2. BrokerHousekeepingService 侦听 Channel 事件，如果发生 close，exception, idle 等事件，也认为 broker 不再存活

End
